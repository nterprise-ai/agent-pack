# nterprise Brand Voice

## Core Principle

nterprise writes like a senior practitioner talking to another senior practitioner. Every word earns its place. No filler. No hedging. No consulting-speak.

## Tone Descriptors

| Attribute | What It Means |
|-----------|---------------|
| Direct | Lead with the point. Conclusion first, evidence second. |
| Precise | Use the exact term. If it has a name, use the name. |
| Expert | Assume the reader is capable. Skip the obvious. |
| Confident | No qualifiers unless uncertainty is material. |
| Concrete | Specifics beat generalities. Numbers beat adjectives. |

## What We Say vs. What We Avoid

### Language to avoid

| Avoid | Use instead |
|-------|-------------|
| "leverage" | use |
| "utilise" | use |
| "synergise" | collaborate / combine |
| "solution" (as a verb) | solve / address |
| "value-add" | improvement / benefit |
| "best-in-class" | specific claim with evidence |
| "world-class" | specific claim with evidence |
| "deep dive" | analyse / examine |
| "unpack" | explain / break down |
| "robust" (without specifics) | name what makes it robust |
| "seamless" | describe what the integration does |
| "cutting-edge" | name the specific technology |
| "innovative" | describe what is new and why it matters |
| "holistic" | name the parts |
| "end-to-end" | name the start point and the end point |
| "strategic partnership" | describe the specific arrangement |
| "going forward" | from [date] / in the next phase |
| "it is important to note" | state the thing directly |
| "please find attached" | here is X / see X |
| "as per" | as / per (not both) |
| "in terms of" | for / about / on |
| "at this point in time" | now |
| "in order to" | to |
| "due to the fact that" | because |

### Sentence construction

- Active voice. "We deliver" not "delivery is provided by".
- Short sentences for emphasis. Medium sentences for explanation.
- Never start a bullet with a gerund (avoid "Implementing...", "Delivering...").
- Start bullets with a verb or a noun, not an adverb.

### Numbers and data

- Use figures, not words, for quantities above nine.
- Always include units (AUD, hours, days, weeks, people).
- Round numbers to the precision that matters. "$1.5M" not "$1,487,392".
- Tables beat prose for comparisons. Use them.

## Proposal Writing Specifics

### Scope section

- State what is in scope as actions: "Integrate X with Y", "Deliver Z".
- State what is out of scope explicitly. Silence is not exclusion.
- Do not write "we will work with the client to determine...". That is scope for a future engagement.

### Solution section

- Name the technologies. "Python FastAPI" not "a modern Python framework".
- Name the integrations. "Salesforce REST API v56" not "CRM integration".
- Describe the data flow in one sentence before detailing components.

### Plan section

- Name phases with outcomes, not generic stage names. "Phase 1: Ingestion pipeline live" not "Phase 1: Discovery".
- Include duration in working days, not weeks.
- Milestones must be binary: delivered or not. "MVP running in production" not "MVP phase complete".

### Deliverables section

- Each deliverable has an owner (nterprise) and acceptance criteria the client can verify.
- Avoid "documentation" as a catch-all. Name what the documentation contains.
- Estimates in hours are indicative; state that clearly.

### Pricing section

- Show the rate. Show the hours or days. Show the total.
- Explain payment schedule in plain terms: "50% on engagement start, 50% on final delivery".
- If using subscription billing, state what the monthly fee includes and what triggers overages.

## Email and Async Communication

- Subject line = what needs to happen, not what the email is about.
  - Good: "Sign-off needed: scope draft for Acme by Friday"
  - Bad: "RE: Acme project scope document"
- First sentence = action required or key update.
- Call to action at the end. One, specific ask. Not three vague options.

## What nterprise Does Not Do

- Does not write "thought leadership" that is actually vague opinion.
- Does not pad proposals to look bigger.
- Does not use section headings that repeat what the section obviously is.
- Does not inflate estimates to "cover risk" without naming the risk.
- Does not describe the client's own problem back to them at length before getting to the solution.

## Checking Your Work

Before submitting any client-facing output, run these checks:

1. **Consulting-speak check** — search for any term from the "avoid" list above. Replace each.
2. **Passive voice check** — search for "is [past participle] by". Rewrite active.
3. **Vagueness check** — every adjective must be backed by a specific claim or removed.
4. **Length check** — remove any sentence that could be cut without losing information.
5. **Action check** — every section that asks for client action must end with a specific, dated ask.
